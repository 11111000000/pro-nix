# Политика агентов pro-nix

Кратко: изменения должны быть воспроизводимыми, минимальными и проверяемыми. Агенты рассматривают репозиторий как совокупность публичных контрактов (SURFACE) и реализаций (HOLO, модули).

## Канонический порядок работы
0. Перед правкой предложить имя ветки и создать workspace в git: `feat/<short-desc>` → `git checkout -b feat/<short-desc>`.
1. Ознакомиться с `AGENTS.md`, `SURFACE.md`, `HOLO.md`, `README.md`.
2. Сформулировать Intent — одна доминирующая цель на одно изменение.
3. Оценить влияние на публичное поведение (SURFACE). При изменении обновить `SURFACE.md` и указать Proof.
4. Внести минимальные изменения в код/конфигурацию, следуя существующим паттернам.
5. Прогнать Verify (локальные проверки и тесты).

Примечание: перед созданием коммита агент уточняет у пользователя — закоммитить сейчас или показать diff для ревью.

## Change Gate (коротко)
- Intent: краткая цель.
- Pressure: Bug | Feature | Debt | Ops.
- Surface impact: перечислить записи SURFACE и их статус.
- Proof: точные команды/тесты, подтверждающие поведение.
- Migration: требуется при изменении [FROZEN] записей.

Правило отказа: если изменение трогает [FROZEN] элемент SURFACE без Proof — остановиться и запросить данные.

## Правила оформления и стиля
- Язык: русский.
- Стиль: краткие контракты (цель, эффекты, проверки), не «пожелания».
- Документы и docstring — информативные, без излишних рассуждений.

## Технические рекомендации (суть)
- Nix: вносить вклад, не финализировать; предпочитать `lib.mkDefault`; избегать скрытой зависимости на `config.environment.systemPackages`.
- Emacs Lisp: одна функция — одна задача; публичные функции — docstring; изменения — с ERT-тестом.
- Структура кода: один файл — одна ответственность; не смешивать экспериментальный код с каноническим.

## Критерии качества
- Минимум изменений, закрывающий Intent.
- Предпочитать существующие паттерны репозитория.
- Указывать, как проверять (какие команды/тесты) и почему правка безопасна.

## Управление изменениями
- Большие правки разбивать на этапы: Surface → Proof → Code → Verify.
- При мультисистемной правке сузить Intent и разбить работу на шаги.

## Быстрые проверки
- `./tools/surface-lint.sh`
- `./tools/holo-verify.sh`
- `nix fmt`
- `nix flake check`
- headless ERT (для Emacs)

## Конфликты и сомнения
- При сомнении следовать стилю репозитория и минимизму.
- Если не ясно — спросить Intent у автора задачи.
