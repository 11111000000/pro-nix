# План: Гармоничный Emacs для pro-nix

## Цель
Собрать простую, надёжную и приятную Emacs-систему для `pro-nix`, в которой:
- Nix отвечает за среду исполнения и пакеты,
- Org отвечает за глобальные клавиши,
- модули отвечают только за поведение,
- опциональные улучшения включаются явно,
- вся система остаётся понятной с первого чтения.

## Диалектическая критика текущего состояния

### 1. Слишком много механизмов вокруг одной простой задачи
Ранее план пытался решить сразу всё:
- миграцию из `~/pro`,
- регистрацию предложенных клавиш,
- импорт/экспорт ключей,
- provenance,
- специальные key-utils,
- отдельные сценарии для EXWM, tabs, terminals.

Это полезно лишь тогда, когда центр уже стабилен.
Пока центр не закреплён, дополнительные механизмы делают систему не умнее, а шумнее.

### 2. Глобальные клавиши не должны иметь второй жизни
Если идея уже ясна: все глобальные клавиши живут в Org-файле, то второй путь
для тех же клавиш через Lisp-хелперы, регистрацию suggestions и импортёры
разрушает ясность.

Одна сущность — один путь.

### 3. Не всё из `~/pro` нужно переносить как "систему"
В `~/pro` есть зрелые, полезные вещи. Но там же есть и личная инерция:
- накопленные обходные пути,
- тяжёлые UX-слои,
- решения под один компьютер и один ритм работы.

Если всё это перенести механически, `pro-nix` станет не переносимым профилем,
а копией одного частного быта.

### 4. Красота требует строгого центра
Гармония здесь достигается не количеством фич, а чётким разделением:
- Nix: пакеты, бинарники, runtime, внешние утилиты.
- Emacs Lisp: поведение модулей.
- Org: глобальные клавиши.
- Пользователь: локальные overrides и opt-in модули.

## Принцип гармонии

### Инвариант
То, что должно быть одинаково везде:
- глобальные клавиши только через `emacs-keys.org` и `~/.config/emacs/keys.org`
- модульный loader
- Nix-first поставка пакетов
- opt-in для тяжёлых UX-слоёв
- отсутствие top-level `global-set-key` вне `keys.el`

### Переменное
То, что может отличаться от машины к машине:
- EXWM включён или нет
- tabs нужны или нет
- terminal helpers нужны или нет
- GUI/TTY поведение

### Личное
То, что принадлежит пользователю:
- `~/.config/emacs/modules.el`
- `~/.config/emacs/modules/*.el`
- `~/.config/emacs/keys.org`

## Архитектура прекрасной системы

### 1. Nix-first ядро
Всё важное должно приходить из Nix.

Обязательное ядро:
- `consult`
- `vertico`
- `orderless`
- `marginalia`
- `corfu`
- `cape`
- `consult-dash`
- `consult-eglot`
- `consult-yasnippet`
- `yasnippet`
- `avy`
- `expand-region`
- `projectile`
- `treemacs`
- `vterm`
- `ace-window`

Системные утилиты:
- `ripgrep`
- `fd`
- `findutils`

Опционально:
- `golden-ratio`
- `multi-vterm`
- внешний `pro-tabs`, если он станет нормальным Nix-пакетом

### 2. Org-first глобальные клавиши
Глобальные клавиши не обсуждаются в модулях.

Они существуют только здесь:
- `repo/emacs-keys.org`
- `~/.config/emacs/keys.org`

`keys.el` — единственный загрузчик этой политики.

### 3. Модульный Emacs
Каждый модуль должен быть маленьким и честным:
- если пакет есть — включить поведение,
- если пакета нет — тихо отступить,
- не назначать глобальные клавиши,
- документировать opt-in переменные.

### 4. Опциональные UX-слои
Три слоя разрешены как модули:
- `terminals.el`
- `windows.el`
- `pro-tabs.el`

Они не должны превращаться в автономные мини-фреймворки.

## Что реально важно перенести из `~/pro`

### Приоритет 1 — уже стоит в центре
- `consult` helpers: `pro/consult-find`, `pro/consult-buffer`
- completion stack: `corfu`, `cape`, `consult-*`
- org-based key loading

### Приоритет 2 — нужно довести до ясности

#### Терминалы
Из `~/pro/интеграция/про-терминалы.el` действительно важно:
- `vterm` как основной терминал
- корректный `yank` в терминал
- `interrupt` helper
- вменяемое поведение copy-mode

Неважно прямо сейчас:
- большой набор vterm-specific хитростей
- сложная тема vterm
- глубокая eshell-эстетика

Вывод:
- `terminals.el` должен остаться минимальным модулем.
- Если появится подтверждённая боль, добавляются конкретные helper-функции,
  а не переносится весь старый пласт.

#### Вкладки
Из `~/pro/среда/про-внешний-вид.el` важно не существование `pro-tabs`,
а поведение:
- `tab-bar` как основа
- `tab-line` как локальное дополнение
- optional icons

Вывод:
- если внешний `pro-tabs` не живёт в Nix нормально, built-in `tab-bar/tab-line`
  — это правильный центр.
- `pro-tabs.el` должен быть лишь thin wrapper, не идеология.

#### Окна
Из `~/pro/интерфейс/про-окна.el` важно:
- `winner-mode`
- `ace-window`
- лёгкий `display-buffer-alist` для терминалов и служебных окон

Неважно прямо сейчас:
- тяжёлый popper/popwin слой
- сложная экосистема popup-management

Вывод:
- `windows.el` должен быть маленьким и предсказуемым.

### Приоритет 3 — позже
- EXWM simulation keys и hooks
- tray/system buffer polish
- vterm theme
- fancy tab icons

## Простой roadmap

### Фаза A — центр уже должен быть ясен
- завершить Nix-first список пакетов
- завершить `emacs-keys.org`
- поддерживать `keys.el` как единственный global-key loader

### Фаза B — минимальные модули
- `terminals.el`
- `windows.el`
- `pro-tabs.el`

### Фаза C — только нужные поведенческие мелочи
- маленький `display-buffer-alist` для `vterm` и `*Messages*`
- локальные mode-specific bindings

### Фаза D — EXWM only after center is calm
- simulation keys
- rename hooks
- safe session glue

## Критерий красоты
Конфиг прекрасен, если:
- его можно объяснить за пять минут,
- в нём нет второго способа сделать то же самое,
- всё важное ставится из Nix,
- глобальные клавиши живут только в Org,
- тяжёлые слои включаются одной переменной,
- CI проверяет простые и важные инварианты.

## Практический следующий шаг
1. Довести до конца Nix-first список.
2. Оставить `terminals.el`, `windows.el`, `pro-tabs.el` как small wrappers.
3. Добавить маленький `display-buffer` policy.
4. Потом возвращаться к EXWM.

### Что проверено после переключения (runtime checklist)

- `nixos-rebuild switch` выполнен и конфигурация активирована: см. системный путь в /nix/store (rebuild artefact).
- Headless Emacs E2E пройден локально на хосте: выполнено `./scripts/dev-emacs-pro-wrapper.sh --batch -l scripts/test-emacs-e2e-assertions.el` — все обязательные пакеты доступны и функции работают (vertico, consult, corfu, cape, vterm, winner, ace-window, embark, marginalia).
- CI workflow добавлен: `.github/workflows/emacs-e2e.yml` запускает `scripts/lint-keys.sh` и headless E2E в reproducible Nix shell.
- Генератор списка пакетов `scripts/generate-provided-packages.el` добавлен — используйте его чтобы обновить `~/.config/emacs/provided-packages.el` или репозиторный fallback `emacs/base/provided-packages.el`.

Примеры команд для оператора:

- Перегенерировать список Nix-пакетов локально:
  - emacs --batch -l scripts/generate-provided-packages.el --eval '(generate-provided-packages "nix/provided-packages.nix")'
- Прогнать headless E2E локально:
  - ./scripts/emacs-pro-wrapper.sh --batch -l scripts/emacs-e2e-assertions.el
- Запустить линтер клавиш:
  - ./scripts/lint-keys.sh
