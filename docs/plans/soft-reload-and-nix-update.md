**Soft Reload And Nix Update**

Цель: дать пользователю возможность плавно обновлять конфигурацию Emacs и пакеты без полной перезагрузки Emacs, а также аккуратно применять изменения Nix-окружения к уже запущенному Emacs.

1) Механизм мягкой перезагрузки конфигурации (внутри Emacs)
- Компоненты:
  - pro/reload-module (emacs/base/modules/reload.el): перезагружает один модуль по имени (load-file).
  - pro/reload-all-modules: итерирует по pro-emacs-base-default-modules и загружает каждый модуль.

- Ограничения и инварианты:
  - Модули должны быть idempotent (без сайд-эффектов при повторной загрузке) для корректной работы pro/reload-*.
  - Состояние (сессии, открытые процессы) не восстанавливается автоматически; модули, которые создают stateful объекты, должны экспонировать функции migrate/reset.

- Как использовать:
  - В интерактивном Emacs: M-x pro/reload-module RET terminals RET — перезагрузить только terminals.
  - M-x pro/reload-all-modules — попытаться мягко перезагрузить весь набор модулей.

2) Обновление пакетов MELPA в фоне
- Компоненты:
  - scripts/melpa-update.el — batch-скрипт для Emacs, который обновляет archives и устанавливает/обновляет пакеты.
  - pro/update-melpa-in-background (emacs/base/modules/reload.el) — стартует фоновой Emacs-процесс, который запускает scripts/melpa-update.el.

- Поведение:
  - В интерактивном Emacs: M-x pro/update-melpa-in-background — запускает `emacs --batch -Q -l scripts/melpa-update.el` в фоновом процессе; прогресс пишется в буфер *pro-melpa-update*.
  - После завершения можно вызвать pro/reload-module для перезагрузки конкретных модулей, если это требуется.

3) Плавное обновление Nix-окружения для запущенного Emacs
- Проблема: Nix предоставляет системные пакеты и Elisp-бандлы в /nix/store. Когда собран новый набор пакетов, Emacs нужно заставить использовать новые site-lisp пути без явного перезапуска.
- Возможные подходы:
  - Рекомендация 1 (простая, надёжная): Перезапустить Emacs (наиболее безопасно). Можно автоматизировать: скрипт сохраняет состояние (open-buffers, window layout) и рестартует Emacs, затем восстанавливает состояние.
  - Рекомендация 2 (мягкий): обновить `load-path` и переинициализировать некоторые пакеты:
    1. Новой Nix сборкой получаем список site-lisp путей (скриптом). 2. В интерактивном Emacs выполнить: (setq load-path (append '<new-paths> (remove old-paths load-path))) и пере-загрузить модули/provide нужные пакеты. Это рискованно для пакетов с C-расширениями (libvterm) — их нельзя «подменить» без перезапуска Emacs.

- Предложенный процесс для про-никс:
  - Автоматический path-refresh: скрипт `scripts/nix-update-emacs-paths.sh` собирает новые /nix/store/*/share/emacs/site-lisp и генерирует файл emacs/base/provided-packages.el или просто возвращает список путей.
  - В Emacs: команда `pro/nix-refresh-load-path` (не реализована здесь) которая читает список новых путей и подставляет их в `load-path` и `default-frame-alist` при необходимости.
  - Для C-расширений (vterm) — всегда делать рекомендованный рестарт Emacs; можно показать пользователю подсказку "Restart Emacs to apply native module upgrades".

4) Риски и рекомендации
- Всегда делать резервную копию emacs-keys.org и других важных файлов перед авто-операциями.
- Для безопасной работы полагаемся на idempotent загрузку модулей; если модуль держит state, добавить функцию `pro/<module>-reset` чтобы корректно сбросить state перед reload.
- Для production: предпочтительнее автоматизированный рестарт с state save/restore.

5) Как мы это внедрим в pro-nix
- Добавить `emacs/base/modules/reload.el` (готово) и `scripts/melpa-update.el` (готово).
- Добавить небольшую документацию и примеры использования (этот файл).
- Поэтапно реализовать `pro/nix-refresh-load-path` и `scripts/nix-update-emacs-paths.sh` (следующий шаг).
