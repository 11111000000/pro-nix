# Peer cache (локальный обмен Nix артефактами)

Цель
-----

Сделать так, чтобы машины из семейства `pro-nix` в одной локальной сети могли оппортунистически делиться уже скачанными или собранными Nix-артефактами. Мы избегаем постоянного always‑on сервера и вместо этого используем простую, надёжную модель: обнаружение соседей + передача store-путей по SSH. Это минимально по операции и даёт практический выигрыш времени и трафика в условиях динамических IP и меняющихся ролей машин.

Коротко — философия
---------------------

- Тезис: если одна машина уже скачала/собрала — давайте использовать это богатство локально.
- Антитезис: не делаем сложной централизованной инфраструктуры там, где достаточно простого контракта доверия и обмена.
- Синтез (путь Дао): минимальные, идемпотентные действия — SSH для доставки, mDNS/Avahi для обнаружения, простые правила приёма и GC. Делай мало, но делай безопасно.

Компоненты решения
-------------------

- SSH — канал доставки `/nix/store` путей между машинами: `nix copy --to ssh://user@peer /nix/store/…`.
- mDNS / Avahi — обнаружение соседей в локальной сети (работает при DHCP/dynamic IP): `peer.local` вместо хардкод IP.
- Простые скрипты — wrapper, который после успешной сборки отправляет пути соседям; и лёгкий systemd-user unit для запуска по таймеру или после сборки.
- Правила приёма — на принимающей стороне: ограничение размера/возраста, регулярный `nix-collect-garbage`.

Почему этот путь практичен
-------------------------

- Не требует всегда‑он хоста — подходит для ноутбуков, у которых роли меняются.
- Учитывает динамические IP с помощью mDNS/Avahi.
- Минимально вмешивается в Nix‑парадигму: Nix остаётся главным валидатором содержимого; мы лишь копируем store‑пути.
- Позволяет эволюционировать в локальный proxy или подписной cache при появлении выделенного сервера.

Техническое руководство (быстрый старт)
-------------------------------------

1) Требования на каждой машине

- Nix установлен и работает.
- SSHd включён и доступен (systemd сервис `sshd` / пакет `openssh`).
- Avahi (mDNS) установлен и служит (`avahi-daemon`).
- Достаточно места в `/nix/store` на машинах‑приёмниках.

2) Аутентификация

- Обменяйтесь публичными SSH-ключами между участниками. Поместите публичные ключи доверенных участников в `~/.ssh/authorized_keys` на каждой машине, которой вы доверяете принимать артефакты.
- Опция безопасности (рекомендуется для дисциплины): в `authorized_keys` можно фиксировать forced-command, ограничивающий ключ конкретной командой. Пример (на принимающей машине):

```text
command="/usr/local/bin/nix-receive-guard.sh" ssh-rsa AAAA... user@host
```

Эта команда должна проверять входящий канал и разрешать `nix copy` только из ожидаемых источников. (Пример скрипта мы приводим в разделе инженерии.)

3) Discovery (обнаружение соседей)

- Самый простой путь: поддерживать `~/.local-peers` — файл с именами хостов вида `cf19.local` и `huawei.local` (по одному в строке).
- Автоматический путь: использовать avahi/bonjour для поиска рабочих станций. Пример команды для поиска `._workstation._tcp`:

```bash
avahi-browse -rt _workstation._tcp
```

или получить адрес конкретного сервиса:

```bash
avahi-resolve -n cf19.local
```

4) Push-модель: скрипт отправки

Скрипт `scripts/push-nix-to-peers.sh` (в репозитории) — берёт список путей и копирует их к peer'ам из `~/.local-peers`. Скрипт идемпотентен: если путь уже есть у peer, nix пропустит копирование. Пример использования:

```bash
# Передаём store-пути, полученные после сборки
./scripts/push-nix-to-peers.sh /nix/store/abcd... /nix/store/xyz...
```

5) Wrapper: автоматический запуск после сборки

Чтобы не запускать вручную, можно обернуть `nix build`/`nix-build` в wrapper `scripts/nix-build-and-share.sh`. Wrapper запускает build, извлекает созданные пути и вызывает push‑скрипт.

6) Systemd‑user unit (опционально)

Можно поставить `systemd --user` сервис и таймер, который раз в сутки/при логине пытается синхронизовать недавно созданные пути соседям. В docs мы приводим пример unit и timer.

7) Политика приёма и GC

- На принимающей стороне держите правило: не принимать пути, превышающие X GB, или старше Y дней. Это можно наложить в `nix-receive-guard.sh`.
- Плановый `nix-collect-garbage -d` запускайте по расписанию (timer) или при достижении порога использования диска.

8) Как проверить вручную

- На машине‑доноре: `nix copy --to ssh://user@peer.local /nix/store/<path>`
- На принимающей: `nix path-info /nix/store/<path>` — должен показать информацию о пути.

Безопасность и доверие
----------------------

- Базовая модель доверия — SSH авторизация. Это достаточно для домашней сети и небольших команд.
- Для более строгой модели используйте подписанные кэши (Cachix / подписной self‑hosted cache). Переход на них — естественная следующая стадия.

Инженерные примеры (скрипты и unit)
-----------------------------------

- `scripts/push-nix-to-peers.sh` — примерный рабочий скрипт (в репозитории). Он использует `~/.local-peers` по умолчанию, поддерживает avahi discovery fallback и логирует ошибки.
- `scripts/nix-build-and-share.sh` — wrapper, запускает `nix build` и выполняет `nix path-info`/парсинг результатов для последующего пуша.
- Пример systemd user unit и timer приведены в `docs/plans/peer-cache-plan.md`.

Когда переходить к следующему шагу
---------------------------------

- Если hit-rate в локальной сети высокий и у вас появился хотя бы один часто‑включённый хост, подумайте о ncps/nginx или full signed cache (cachix/harmonia). Тогда отдача вырастет, а администрирование централизуется.

Философская ремарка
-------------------

Дао простоты: не заставляй сеть быть тем, чем она не является. Пусть каждая машина делает минимум — делиться своими трудами через простые, надёжные каналы. Пусть это будет достаточным началом; если поток станет мощнее — строй плотины. Но сначала — вода должна течь свободно.
