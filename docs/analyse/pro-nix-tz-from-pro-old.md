# pro-nix: развёрнутое ТЗ по мотивам pro.old

## 1. Тезис

`pro.old` полезен не как шаблон для копирования, а как источник проверенных интуиций.

Главная ценность старого конфига:

- Emacs рассматривается как рабочая среда, а не как набор пакетов
- UI, поиск, клавиши, Org, язык и проект должны быть связаны в одну систему
- визуальная среда должна быть удобной для русского текста и кода
- пользователь должен иметь возможность переопределять системные решения

Но переносить это надо не буквально, а заново, в стиле `pro-nix`.

## 2. Антитезис

Старый конфиг страдает от накопления:

- слишком много всего в одном месте
- слишком много декоративных решений
- слишком много комментариев, объясняющих сам код вместо смысла
- слишком много частных зависимостей и экспериментальных подсистем

Если это перенести без фильтра, новый конфиг станет тем же самым, только под другим именем.

Нам это не нужно.

## 3. Синтез

`pro-nix` должен быть:

- минимальным по механике
- богатым по смыслу
- ясным по структуре
- пользовательски переопределяемым
- литературно оформленным

То есть:

- код не как «мертвый конфиг», а как статьи об аспектах среды
- каждая часть отвечает за один аспект
- все кастомные переменные и функции называются только с префиксом `pro-`
- в именах переменных и функций не использовать кириллицу
- комментарии и пояснения писать на русском

## 4. Что именно переосмысливаем

### 4.1. Визуальная среда

Не переносить чужие настройки как есть.

Нужно заново собрать собственный визуальный язык:

- шрифт для кода с поддержкой лигатур, если это действительно помогает читать код
- шрифт интерфейса, если нужно отделить UI от кода
- масштабирование шрифта как ежедневная операция
- качественная поддержка кириллицы в Emacs, TTY и desktop-окружении
- иконки только там, где они ускоряют распознавание
- табы, модель строки и completion UI должны быть спокойными и читаемыми

Диалектика:

- тезис: внешний вид повышает удобство
- антитезис: внешний вид легко превращается в шум
- синтез: только та красота, которая повышает ясность

### 4.2. Навигация и поиск

Нужны не просто пакеты, а единая траектория поиска:

- `consult`
- `vertico`
- `orderless`
- `marginalia`
- поиск по проекту
- переход к символам, xref, imenu
- быстрый доступ к буферам и файлам

Диалектика:

- тезис: много способов найти нужное
- антитезис: много способов создают разнобой
- синтез: один стек поиска и одна логика поведения

### 4.3. Клавиши

Ключевая идея: клавиши должны быть внешним слоем, а не жестко прошитым кодом.

Требования:

- системные дефолты задаются в `pro-nix`
- пользователь может заменить их через `~/.emacs.d/keys.org`
- формат `keys.org` должен быть удобным, лучше всего табличным Org-форматом
- загрузчик должен быть простым и предсказуемым

Диалектика:

- тезис: клавиши должны быть удобны сразу
- антитезис: клавиши должны быть изменяемы без переписывания конфига
- синтез: дефолт + пользовательское переопределение

### 4.4. AI: aitunnel и openrouter

Это не каталог моделей, а политика доступа.

Нужно:

- минимальная интеграция `gptel`
- поддержка `aitunnel`
- поддержка `openrouter`
- один центр выбора backend-а
- один компактный список предпочтений моделей

Не нужно:

- огромные каталоги моделей
- таблицы цен внутри runtime-логики
- бесконечная гонка за новыми названиями

Диалектика:

- тезис: доступ к сильным моделям важен
- антитезис: рынок моделей быстро устаревает
- синтез: политика выбора, а не энциклопедия

### 4.5. Org-mode как рабочая форма

Org должен быть не заметкой, а структурой работы.

Нужно:

- удобная работа с таблицами
- удобные source blocks
- подсветка и визуальная ясность
- полезные capture/link/navigation привычки
- интеграция с кодом, планами, анализом и настройками

Особый акцент:

- Org-таблицы должны годиться для `keys.org`
- Org должен служить для ТЗ, анализа, записей и конфигурации

### 4.6. Языки программирования

Поддержка должна быть не «для галочки», а для реальной работы.

Приоритет:

1. Lisp
2. Nix
3. Python
4. C
5. Java
6. Haskell

Для каждого языка нужно брать только самое полезное:

- syntax/indent
- LSP/eglot только если приносит реальную пользу
- REPL/eval там, где это естественно
- навигацию по символам и проектам

### 4.7. Git и проектная работа

Нужен не «git-зоопарк», а ясная рабочая траектория:

- `magit`
- project-aware navigation
- xref/imenu
- diff/navigation helpers только если они реально ускоряют работу

Для проектов важно:

- быстро понимать корень проекта
- быстро переходить по буферам и символам
- видеть статус изменений без перегруза

## 5. Стиль исходников

Это обязательная часть ТЗ.

### 5.1. Общий стиль

Все исходники должны быть оформлены как статьи о рабочей среде:

- заголовок
- краткая постановка задачи
- комментарий о смысле секции
- код
- заключение секции

Не как сухой конфиг, а как осмысленное описание среды.

### 5.2. Комментарии

- все комментарии на русском
- комментарии должны объяснять смысл, а не повторять строку кода
- допускаются короткие содержательные пояснения перед блоком
- избегать многословия ради многословия

### 5.3. Имена

- все пользовательские функции и переменные начинаются с `pro-`
- не использовать кириллицу в именах переменных и функций
- внешние пакеты можно называть как они есть, но обёртки и свои слои только `pro-*`

### 5.4. Разделение слоёв

Нужно разделять:

- базу Emacs
- UI
- навигацию
- клавиши
- Org
- языки
- git/project
- AI
- TTY/desktop-нюансы

## 6. Архитектурное требование

`pro-nix` должен поддерживать такой принцип:

- системный слой задаёт разумные дефолты
- пользовательский слой может переопределить почти всё
- загрузка должна быть устойчивой
- optional-пакеты не должны ломать базовый запуск

Идея: сначала устойчивость, потом красота, потом расширения.

## 7. Что переносить из pro.old, а что нет

### Переносить как пример

- принцип модульности
- поиск и навигацию
- `shaoline`, `pro-tabs`, `consult`, `vertico`, `orderless`, `marginalia`
- полезные куски Org-эргономики
- полезные куски Git/project-навигации
- полезные куски UI-поведения

### Не переносить буквально

- длинные декоративные комментарии
- дублирующие подсистемы
- неустойчивые экспериментальные настройки
- слишком крупные таблицы моделей и цен
- частные пути и локальные зависимости

## 8. Приоритеты внедрения

1. Базовая форма исходников и именование
2. UI: шрифт, кириллица, масштаб, иконки
3. Клавиши и `keys.org`
4. AI-политика `aitunnel` / `openrouter`
5. Поиск и навигация
6. Org-mode
7. Языки программирования
8. Git/project-workflow
9. Шлифовка визуальных деталей

## 9. Структура модулей `pro-nix`

Ниже не готовый код, а правильная карта разбиения.

### 9.1. База загрузки

#### `emacs/base/early-init.el`

Назначение:

- ранняя безопасность загрузки
- отключение автозапуска пакетов
- минимальные параметры фрейма
- подготовка к безопасному старту Emacs 30+

Сюда не класть:

- UI-слои
- языковые настройки
- проектную логику

#### `emacs/base/init.el`

Назначение:

- входная точка Emacs
- запуск базового сайта
- загрузка модулей по списку

Это должен быть очень короткий файл.

#### `emacs/base/site-init.el`

Назначение:

- поиск пользовательских модулей
- приоритет пользовательского слоя над системным
- загрузка списка модулей по `pro-emacs-modules`

Это центральный диспетчер, но не место для тяжёлой логики.

### 9.2. UI и текст

#### `emacs/base/modules/ui.el`

Назначение:

- шрифты
- масштабирование
- иконки
- модель строки
- вкладки
- визуальные лица
- спокойный и читаемый интерфейс

Что ожидается внутри:

- настройки кириллицы
- Fira-лигатуры или их корректный аналог
- `shaoline`
- `pro-tabs`
- `kind-icon`
- `treemacs-icons-dired`
- `nerd-icons-ibuffer`
- `cursor-chg`
- `display-line-numbers` только для кода

#### `emacs/base/modules/text.el`

Назначение:

- общие текстовые удобства
- структурная читаемость кода и документов
- подсветка пар
- полезные визуальные мелочи для редактирования

Что ожидается внутри:

- парные скобки
- подсветка уровней
- `prettify-utils` или минимальная аналогия
- `eldoc-box`
- `imenu-auto-rescan`

### 9.3. Навигация и поиск

#### `emacs/base/modules/nav.el`

Назначение:

- `consult`
- `vertico`
- `orderless`
- `marginalia`
- xref
- imenu
- project buffer navigation
- быстрый поиск по файлам

Это должен быть один цельный навигационный слой.

### 9.4. Клавиши

#### `emacs/base/modules/keys.el`

Назначение:

- загрузка пользовательских клавиш из `~/.emacs.d/keys.org`
- системные дефолты
- простое переопределение пользователем

Требование:

- формат должен быть читаемым как таблица Org
- loader не должен быть сложным
- если пользовательский файл есть, он главнее

### 9.5. Org-mode

#### `emacs/base/modules/org.el`

Назначение:

- основные org-удобства
- source blocks
- таблицы
- capture/link/navigation
- визуальная читаемость
- использование Org как формата для ТЗ и настроек

Особое требование:

- этот модуль должен дружить с `keys.org`
- таблицы должны быть удобны для ежедневного редактирования

### 9.6. Языки

#### `emacs/base/modules/lisp.el`

Назначение:

- Lisp как главный язык конфигурации
- удобство редактирования Emacs Lisp
- поддержка структурного редактирования

#### `emacs/base/modules/nix.el`

Назначение:

- `nix-mode`
- минимальные Nix-удобства
- переход к сборке системы

#### `emacs/base/modules/python.el`

Назначение:

- Python как основной scripting-слой для org-babel и утилит
- запуск, навигация, LSP при необходимости

#### `emacs/base/modules/c.el`

Назначение:

- C как системный язык
- минимальный, но надёжный набор удобств

#### `emacs/base/modules/java.el`

Назначение:

- Java как поддерживаемый язык проектов
- только полезный минимум

#### `emacs/base/modules/haskell.el`

Назначение:

- Haskell как опциональный, но полноценный язык
- подключать только если toolchain реально нужен

### 9.7. Git и проект

#### `emacs/base/modules/git.el`

Назначение:

- `magit`
- git workflow
- быстрый переход к статусу и изменениям

#### `emacs/base/modules/project.el`

Назначение:

- project-root логика
- project-aware navigation
- интеграция с `consult` и `xref`

### 9.8. AI

#### `emacs/base/modules/ai.el`

Назначение:

- `gptel`
- `aitunnel`
- `openrouter`
- минимальная политика выбора backend-а

Правило:

- это модуль политики доступа, а не каталог моделей

### 9.9. Интеграции по рабочим сервисам

#### `emacs/base/modules/feeds.el`

Назначение:

- `elfeed`
- чтение и обработка лент

#### `emacs/base/modules/chat.el`

Назначение:

- `telega`
- мессенджер как часть рабочего процесса

#### `emacs/base/modules/agent.el`

Назначение:

- `agent-shell`
- диалог с агентом через буферы

### 9.10. Терминалы

#### `emacs/base/modules/terminal.el`

Назначение:

- `vterm`
- `eshell`
- полезные shell-удобства
- цвет, ввод, yank, prompt

### 9.11. Параметры и переопределения пользователя

#### `~/.emacs.d/modules.el`

Назначение:

- список модулей, которые пользователь хочет включить
- замена системного набора модулей при необходимости

#### `~/.emacs.d/keys.org`

Назначение:

- пользовательский слой клавиш
- таблица с ключами и командами
- правка без касания Nix-конфига

## 10. Связка с Nix

Nix должен только доставлять среду, но не раздувать смысл.

Нужно предусмотреть:

- установку нужных Emacs-пакетов
- шрифты и иконки
- `home.file` для `early-init.el`, `init.el`, `site-init.el`
- `home.file` для `modules.el.example`
- возможность подложить `keys.org`

Если пакет опциональный, Nix не должен ломать запуск без него.

## 11. Проверка качества

Перед тем как считать работу завершённой, проверить:

- все свои функции и переменные начинаются с `pro-`
- кириллицы нет в именах символов
- комментарии на русском
- модули короткие и тематические
- пользовательский слой может заменить системный
- конфиг читается как серия статей, а не как склад настроек

## 12. Итоговая форма

Идеальный `pro-nix` после этой переработки:

- не копирует `pro.old`
- развивает его смысл
- остаётся строгим по архитектуре
- выглядит как живой текст о рабочем окружении
- даёт сильный UI, поиск, клавиши, Org, языки, AI и git-поведение без перегруза

## 9. Критерий готовности

Конфиг считается удачным, если:

- его легко читать как текст о рабочей среде
- его легко переопределять локально
- он не тащит лишнее из старого проекта
- он полезен в ежедневной работе
- он сохраняет философию, а не копирует форму

## 10. Итог

`pro.old` должен стать не источником копирования, а источником разборки на смысловые узлы.

`pro-nix` должен собрать эти узлы заново:

- проще
- чище
- в одном стиле
- с русскими комментариями
- с `pro-*` именованием
- как литературный текст о практике работы

Именно так старое становится полезным, не превращаясь в инерцию.
