Название: Диалектический анализ pro-nix

Краткий вывод
- pro-nix — зрелый репозиторий с практическим набором NixOS/home-manager модулей и Emacs‑ориентированными профилями.
- Сильные стороны: воспроизводимость конфигураций, богатая документация, явная интеграция Emacs/EXWM, понятная модульная структура.
- Проблемы: смешение большого количества vendor-кода в репозитории, недостаток автоматизированных CI-проверок для Nix/Elisp, частые TODO/FIXME и неунифицированные линтеры, недостаточная явная система тестирования конфигураций.

Методология (как делался анализ)
- Прочитал AGENTS.md, SURFACE.md, README и ключевые Nix-модули (modules/*.nix, emacs/*).
- Просканировал репозиторий на .desktop, exwm, gdm, TODO/FIXME, большие директорий с vendor-кодом (analyse/web_modules).
- Сформулировал тезис/антитезис/синтез для ключевых направлений: конфигурация, воспроизводимость, качество кода, тесты и доставляемость.

Диалектическая сводка
- Тезис: pro-nix — набор конфигов и helper-скриптов, которые делают NixOS+Emacs удобным и воспроизводимым. В репо уже есть шаблоны home-manager, session‑скрипты, и хороший набор docs.
- Антитезис: Наличие большого объёма вспомогательного и vendor-кода (web_modules, JS-бандлы), множество TODO/незавершённых скриптов и отсутствие CI делают проект тяжёлым для новых участников и рискованным при изменениях.
- Синтез: Сохранить текущую модульность и богатые шаблоны, но выделить неизменяемый/vendor-код из репозитория, ввести обязательные CI‑шаги и линтеры, добавить воспроизводимые тесты для ключевых Nix-модулей и Emacs-пакетов.

Ключевые наблюдения (конкретно)
1. EXWM-сессия: определяется через home-manager и создаётся в ~/.local/share/xsessions — не всегда видна GDM. (мы добавили системный xsessions файл как обход).
2. modules/ содержит важные defaults (например, cinnamon включён по умолчанию) — это рабочее решение, но может конфликтовать с user-specific профилями; следует вынести опции в host-specific overlay.
3. analyse/web_modules — содержало минифицированные библиотеки; часть vendor-кода была удалена/игнорирована. Если нужно хранить конкретные версии, лучше держать их как flake inputs или отдельные пакеты.
4. Документация обширна, но разделы анализа и планов разрознены; нужен единый index для приоритетов и актуального статуса.
5. Нехватка автоматизации: нет единого CI, нет pre-commit правил и автоформатирования для Nix/Elisp/SH/JS.

Риски
- Риск регрессий при правках Nix-модулей из-за отсутствия тестов.
- Рост репозитория за счёт vendor-кода приведёт к долгому клонированию и сложностям с review.
- Потенциальная рассинхронизация системных и пользовательских xsessions (как было с EXWM), если не стандартизировать размещение сессий.

Рекомендации (кратко)
- 1) Вынести vendor-артефакты (web_modules) в отдельный пакет/артефакт (submodule или отдельный артефакт в nix store) и убрать их из VCS.
- 2) Настроить CI: линтеры (nixpkgs-fmt/nix-linter, shfmt+shellcheck, styler для elisp — byte-compile в CI), unit/ERT тесты для важных elisp-модулей и NixOS test/containers для критичных модулей (desktop, display manager, exwm-session).
- 3) Добавить pre-commit hooks и автоформатирование для поддержания стиля (nixfmt, shfmt, gofmt/ruamel для YAML если есть, markdownlint).
- 4) Создать лёгкий набор smoke-tests: проверка наличия .desktop файлов в /usr/share/xsessions при билд‑этапе, проверка генерации home-manager файлов.
- 5) Консолидация документации: единственный docs/README.md и таблица статусов (TODO → backlog → in progress → done).
