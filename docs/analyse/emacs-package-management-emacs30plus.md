# Анализ: управление Emacs-пакетами в `pro-nix` для Emacs 30+

## Вопрос

Как сделать так, чтобы после установки в `pro-nix` работали нужные пакеты, при этом:
- базовая сборка оставалась предсказуемой;
- пакеты из MELPA / GNU ELPA / NonGNU ELPA были доступны;
- git-пакеты можно было ставить и обновлять прямо из Emacs;
- если версия пакета в Nix устарела, Emacs мог взять более свежую версию без правки Nix;
- ориентир был только на Emacs 30 или выше.

## Наблюдения по текущему репозиторию

- В дереве `pro-nix` нет явной конфигурации `package.el`.
- `emacs/base/init.el` только поднимает loader и вызывает `pro-emacs-base-start`.
- `emacs/base/site-init.el` решает загрузку пользовательских и системных модулей, но не управляет пакетными архивами.
- В `system-packages.nix` уже есть `gptel`, значит часть Emacs-экосистемы поставляется через Nix, а часть живёт в Lisp-слое.
- `README.md` уже трактует текст как контракт и фиксирует, что `emacs-keys.org` является source of truth для биндингов.

Вывод: пакетная политика сейчас не оформлена как отдельный слой. Это и есть источник рассогласования между Nix, MELPA и обновлениями из Emacs.

## Фактическая база из актуальной документации

### `package.el`
- `package-archives` задаёт архивы пакетов.
- По умолчанию Emacs использует GNU ELPA и NonGNU ELPA.
- `package-enable-at-startup` надо выключать в early init, если activation контролируется вручную.
- `package-install` и `package-upgrade` работают для архивных пакетов.
- `package-install-upgrade-built-in` позволяет ставить более новые версии built-in пакетов из архивов, но bulk-upgrade с ним опасен.

### `package-vc`
- В Emacs 30+ есть встроенный путь для установки пакетов из VCS.
- Ключевые точки: `package-vc-install`, `package-vc-upgrade`, `package-vc-checkout`, `package-vc-rebuild`.
- `package-vc-selected-packages` хранит явно выбранные VC-пакеты.
- Это даёт нативный механизм для git-зависимостей без отдельного менеджера.

### MELPA
- MELPA остаётся архивом для up-to-date пакетов из upstream source.
- Это нормальный источник для большинства не-builtin зависимостей, если нужна свежесть.

### `straight.el` и `elpaca`
- Оба решают проблему воспроизводимости и VCS-пакетов.
- Но оба добавляют отдельный bootstrap/manager слой.
- Для уже существующего NixOS-профиля это лишняя вторая система истины, если требование лишь такое: "обновлять из Emacs, когда Nix устарел".

## Диалектический разбор

### Тезис
Nix должен задавать рабочее основание: Emacs 30+, системные утилиты, библиотеки, графическую/TTY-среду.

### Антитезис
Nix-пакеты стареют. Если весь Emacs-стек держать в Nix, то обновление частых Lisp-зависимостей становится медленным и тяжёлым.

### Синтез
Разделить слой базовой платформы и слой пользовательских пакетов:
- Nix поставляет только фундамент и критичные бинарные зависимости;
- Emacs управляет пакетами через `package.el` и `package-vc`;
- свежие версии можно добирать из MELPA или VCS прямо из Emacs;
- если пакет критичен для запуска конфигурации, его можно оставить в Nix, но только как временный мост, а не как единственный канал обновления.

## Рекомендуемая политика для `pro-nix`

### Базовый уровень
Оставить в Nix:
- Emacs 30+
- git
- gnutls / curl / make / pkg-config и т. п., если нужны пакетным менеджерам
- системные зависимости для UI, EXWM, desktop и shell-интеграции

### Emacs-уровень
Использовать:
- `package.el` для архивных пакетов
- `package-vc` для git-пакетов

### Архивы
Минимум:
- GNU ELPA
- NonGNU ELPA
- MELPA

### Правило устаревших Nix-пакетов
Если пакет нужен не как системная зависимость, а как Emacs Lisp-функциональность, он должен устанавливаться в Emacs user layer, а не быть навсегда закреплённым в Nix.

## Риски

- Если оставить пакет в Nix и одновременно управлять им из Emacs, будет двойной источник истины.
- Если разрешить bulk-upgrade built-in пакетов, можно случайно перезаписать то, что лучше оставить встроенным.
- Если `package-enable-at-startup` не отключить в early-init, Emacs может активировать пакеты до загрузки вашего loader-слоя.
- Если package-archives выставить без priorities/pinning, MELPA может переопределить более безопасные источники там, где это не нужно.

## Что нужно уточнить в реализации

1. Какие пакеты должны быть системными навсегда, а какие должны жить в user layer.
2. Нужно ли фиксировать `package-archives` в общем профиле или только для Emacs слоя.
3. Нужен ли отдельный набор pinned packages для стабильности.
4. Должен ли `package-vc-selected-packages` храниться в `custom-file` или в явном Lisp-файле.
5. Как пользователь будет делать bootstrap: через `M-x list-packages`, через собственный helper или через init-time bootstrap.

## Предварительный вывод

Для `pro-nix` на Emacs 30+ наиболее прямой и прагматичный путь: встроенный `package.el` + `package-vc`, с MELPA как third-party archive и Nix только как платформой. Это даёт:
- установку из Emacs;
- обновление из Emacs;
- поддержку git-источников;
- минимизацию Nix-зависимости там, где важна свежесть.

## Промежуточное решение для текущего дерева

- `gptel` должен уйти из `system-packages.nix` и жить в Emacs user layer.
- Команды для пакетов должны быть вынесены в отдельный prefix, не пересекающийся с projectile-style `C-c p`.
- `early-init.el` уже правильно выключает автоматическую package activation, это нужно сохранить.
- Следующим фильтром должен стать список остальных Emacs Lisp пакетов в Nix: оставить только то, что реально является runtime dependency, а не user-level feature.

## Граница ответственности

- Nix: Emacs binary, libraries, CLI tools, system integration.
- Emacs package layer: `gptel`, MELPA packages, VC packages, user-upgradable Lisp.
- Docs: фиксируют, какие пакеты где живут и почему.
