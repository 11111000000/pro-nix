# Диалектический анализ agent tooling

## Тезис

Open-source агенты уже достаточно зрелые, чтобы стать частью базового рабочего контурa:

- `goose` даёт нативный agent shell и ACP-провайдеров
- `aider` даёт git-native правки в существующих репозиториях
- `opencode` сохраняет текущий терминальный агентный путь

Если они доступны сразу, пользователь не теряет время на ручной bootstrap перед первой задачей.

## Антитезис

Есть три напряжения.

1. Не все инструменты есть в nixpkgs как готовые деривации.
2. Некоторые CLI требуют отдельной аутентификации и runtime-config.
3. Лёгкая доступность может ухудшить воспроизводимость, если скрыть lazy install.

Особенно это касается `aider` и `opencode`: их можно поставить в PATH, но реальный backend/клиент может подтянуться только на первом запуске.

## Синтез

Решение должно разделить уровни:

- `system-packages.nix` отвечает только за доступность команд в PATH
- `docs/agents.md` описывает первичную настройку
- `docs/plans/agent-tooling.md` фиксирует policy и границы
- secrets остаются вне Nix

Так сохраняется баланс:

- минимальный friction для входа
- явный контроль над ключами и провайдерами
- документированная оговорка, где командный интерфейс lazy, а где полностью packaged

## Вердикт

Агенты стоит включать сразу, но только как текстово описанный, разделённый по слоям контракт.

Бинарник в PATH не должен притворяться полной воспроизводимостью, если под ним есть lazy bootstrap.
