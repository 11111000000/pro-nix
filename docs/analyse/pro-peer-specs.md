# Pro‑Peer: подробные спецификации безопасности и поведения

Цель этого документа — дать исчерпывающие, проверяемые и воспроизводимые спецификации для режима peer‑сети `pro-nix`.
Документ предназначен для инженеров, администраторов и ревьюеров конфигурации.

Ключевые требования
-------------------

1. LAN discovery: все хосты в общей Wi‑Fi сети должны находиться по имени (mDNS / Avahi).
2. Доверие внутри LAN: упрощённый обмен артефактами и SSH‑доступ между доверенными хостами при помощи авторизованных ключей.
3. Безопасность публичного репозитория: конфиг репозитория не должен содержать приватных ключей или секретов; всё секретное — GPG‑зашифрованное и доставляется по защищённому каналу.
4. Минимизация площади атаки: по умолчанию SSH — только по ключам, без паролей; запрет root; запрет X11/TCP‑forwarding; firewall разрешает SSH только из приватных подсетей.
5. Reachability через NAT: опция Tor Hidden Service для адекватного, безопасного доступа извне без проброса портов; альтернативы — Yggdrasil / WireGuard при наличии control plane.
6. Auditing и восстановление: все изменения ключей и важные события логируются; есть процедура rotate/revoke ключей.

Требуемые гарантии (нефункциональные)
------------------------------------

- Надёжность: синхронизация ключей не должна приводить к downtime SSH.
- Отказоустойчивость: если GPG ключ compromised — процедура revoke должна работать быстро и безопасно.
- Минимальная доверенность: в случае утраты публичного конфигурационного репо доступ к хостам не должен быть возможен без приватных GPG ключей.

Компоненты системы
------------------

- Avahi (mDNS) — локальное имяразрешение.
- OpenSSH — только ключи, forced‑command для sensitive keys.
- GPG‑зашифрованный файл authorized_keys.gpg — источник truth для authorized_keys.
- systemd service + timer — sync/rotate authorized_keys.
- scripts/pro-peer-sync-keys.sh — расшифровка и запись authorized_keys.
- scripts/pro-peer-acceptor.sh — пример forced‑command handler (см. ниже).
- scripts/backup-hiddenservice.sh — шифрованный бэкап HiddenServiceDir.
- опции: Yggdrasil, WireGuard helper — по необходимости.

Threat Model (кратко)
---------------------

- Атакующий имеет доступ только к публичному Git‑репозиторию (содержимое конфигов). Он не имеет доступа к GPG приватным ключам и не имеет SSH приватных ключей доверенных пользователей.
- Атакующий потенциально может увидеть systemd unit names, пути и вспомогательные скрипты.
- Цель атакующего: получить shell‑доступ на хостах или подменить authorized_keys.

Защита/механизмы смягчения
-------------------------

1. Все секреты (authorized_keys.gpg, HiddenService keys, WireGuard private keys) держатся вне публичного репо и доступны только под контролем администраторов.
2. GPG приватный ключ хранится отдельно (рекомендуется на hardware token) и не используется в автоматическом CI без операторного подтверждения.
3. Скрипт синхронизации запускается только как systemd oneshot под root и пишет файл с правами 600 и владельцем root:root.
4. Forced‑command в authorized_keys ограничивает действия, которые может выполнить конкретный ключ.
5. Firewall декларативно ограничивает SSH только к RFC1918 сетям и loopback. Доступ извне идёт через Tor Hidden Service, а не через проброс портов.

Примеры allowed command patterns (forced‑command)
------------------------------------------------

Ниже — рекомендуемые паттерны для forced‑command, которые можно разместить в `authorized_keys` (в зашифрованном виде):

- `command="/usr/local/bin/pro-peer-acceptor.sh --allow-nix-copy",from="192.168.0.0/16",no-pty,no-agent-forwarding,no-X11-forwarding,no-port-forwarding ssh-rsa AAAA...`

Разрешённые команды внутри acceptor:
- `nix-store --add` и `nix-store --register` (используется в Nix copy protocol)
- `nix-store --import` / `nix-store --load-db` если требуется

Важно: конкретный набор команд необходимо согласовать и протестировать; acceptor должен логировать и валидировать всё.

Logging и аудит
---------------

- Все операции синхронизации ключей логируются в `journalctl` (unit pro-peer-sync-keys) и в файл `/var/log/pro-peer-acceptor.log` для forced‑command.
- Настроить logrotate для логов и мониторинг (Prometheus/exporter) по желанию.

GC, disk management
-------------------

- На принимающих машинах рекомендовать периодический `nix-collect-garbage -d` или `nix-collect-garbage --delete-older-than 30d`.
- Настроить alert если `/nix/store` превышает 80% выделенного пространства.

Процедура реагирования на компрометацию
---------------------------------------

1. Если GPG приватный ключ компрометирован: обновить/перегенерировать ключи, зашифровать новый authorized_keys.gpg и запустить синхронизацию на всех хостах; принудительно удалить старые публичные ключи.
2. Ротация SSH ключей: при подозрении — отозвать соответствующие ключи в authorized_keys.gpg и пересинхронизировать.
3. Провести forensics: собрать журналы, сделать snapshot диска, проверить integrity системных файлов.

----------
