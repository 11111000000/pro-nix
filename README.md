# pro-nix: воспроизводимая NixOS-платформа

`pro-nix` — это воспроизводимая конфигурация NixOS, Emacs и операционных
инструментов, собранная как единая система контрактов. Репозиторий описывает не
только список пакетов, а рабочую среду целиком: базовую ОС, пользовательский
профиль, сетевые и приватные сервисы, Emacs-платформу, инструменты агентов и
проверки, которые подтверждают наблюдаемое поведение.

Главная идея проекта: состояние машины должно быть выводом из читаемой,
проверяемой и композиционной спецификации. Если поведение важно для пользователя
или оператора, оно должно быть описано как поверхность, иметь Proof и
воспроизводиться через Nix.

## Суть

`pro-nix` соединяет четыре слоя в один контур:

- `NixOS`: системная политика, хосты, сервисы, пакеты, безопасность и сеть.
- `Emacs`: переносимый профиль редактора как рабочей платформы для разработки,
  LSP, REPL, EXWM, модулей и управляемой перезагрузки.
- `Ops`: проверяемые сценарии для peer-discovery, Tor, key sync, headscale,
  storage, resource limits и диагностики.
- `Agents/LLM`: воспроизводимые entrypoint-ы для агентной работы, исследований и
  интеграции с локальными инструментами.

Проект устроен как контрактная система. `SURFACE.md` фиксирует публичные
гарантии. `HOLO.md` фиксирует инварианты и решения. Код в `modules/`, `hosts/`,
`emacs/`, `scripts/` и `tools/` реализует эти гарантии минимальными
композиционными слоями.

## Принципы

- Воспроизводимость важнее локальной оптимизации: одинаковые входы должны давать
  одинаковую конфигурацию.
- Модуль вносит вклад, а не финализирует всю систему: общие модули используют
  композицию, host-уровень фиксирует окончательную политику.
- Публичное поведение документируется как Surface и проверяется Proof-командами.
- Секреты не являются частью репозитория: в git хранится политика, а не
  чувствительные данные.
- Emacs рассматривается как часть системы, а не как внешний редактор с ручной
  настройкой.
- Агентные и LLM-инструменты входят в тот же проверяемый контур, что и ОС.

## Подсистемы

**NixOS**

Базовая система описана через flake и модули. `configuration.nix` собирает общую
политику, а `hosts/*` добавляют host-specific решения: загрузчик, файловые
системы, аппаратные настройки, включение сервисов и локальные ограничения.

**Emacs**

`emacs/home-manager.nix` предоставляет переносимый профиль Emacs. Он определяет
Nix-пакеты, load-path, GUI-слой, модули по умолчанию и интеграцию с Home Manager.
Модули в `emacs/base/modules/` оформляют редактор как расширяемую платформу с
проверяемой загрузкой и soft reload поверхностью.

**Peer, сеть и приватность**

Модули `pro-peer`, `pro-services`, `pro-privacy` и `headscale` описывают
сетевую базу: NetworkManager, resolved, SSH hardening, Avahi/mDNS,
синхронизацию ключей, Tor client, Tor hidden service, Yggdrasil, WireGuard helper
и headscale control plane. Эти возможности включаются опциями и проверяются
контрактными командами.

**Пакеты и runtime**

`system-packages.nix` хранит общий пакетный слой и разделяет базовые утилиты от
тяжёлых optional-пакетов. Это позволяет держать серверные профили минимальными,
а рабочие станции расширять явно.

**Агенты и LLM**

Репозиторий содержит воспроизводимые entrypoint-ы для LLM-исследований и
агентной работы: `llm-lab`, opencode-интеграцию, `proctl`-спецификацию и
вспомогательные приложения flake. Эти инструменты рассматриваются как часть
операционной среды и подчиняются тем же правилам воспроизводимости.

## Хосты

| Хост | Назначение | Особенности |
|------|------------|-------------|
| `huawei` | Основная laptop/workstation-конфигурация. | systemd-boot, zram/opencode limits, рабочий набор сервисов. |
| `cf19` | Lenovo ThinkPad CF-19 и полевая/операционная конфигурация. | GRUB/BIOS, pro-peer key sync, Tor hidden service for SSH. |
| `vm` | Минимальная виртуальная конфигурация. | Быстрая проверка базовой сборки и изоляция экспериментов. |

Сборка хоста выполняется через flake output:

```bash
nix build .#nixosConfigurations.huawei.config.system.build.toplevel
```

Live-активация применяется только после релевантных проверок:

```bash
sudo nixos-rebuild switch --flake .#huawei
```

## Быстрый старт

1. Клонировать репозиторий:

```bash
git clone git@github.com:11111000000/pro-nix.git
cd pro-nix
```

2. Проверить flake:

```bash
nix flake check
```

3. Собрать системный профиль хоста:

```bash
nix build .#nixosConfigurations.huawei.config.system.build.toplevel
```

4. Войти в devshell:

```bash
nix develop .#devShells.x86_64-linux.default
```

5. Запустить Emacs через проектную обёртку:

```bash
./scripts/emacs-pro-wrapper.sh
```

6. Запустить полный проверочный entrypoint, если нужен широкий smoke:

```bash
nix run .#check-all
```

## Контракты и Proof

`pro-nix` различает код и публичное обещание. Код может меняться свободно, пока
не меняет наблюдаемое поведение. Если меняется наблюдаемое поведение, оно должно
быть отражено в контрактных документах и проверках.

- `SURFACE.md` — реестр публичных контрактов: имя, стабильность, спецификация,
  proof-команда.
- `HOLO.md` — инварианты, архитектурные решения, смысл FROZEN/FLUID поверхностей
  и список ключевых проверок.
- `AGENTS.md` — правила работы агентов и инженерные ограничения проекта.
- `CONTRIBUTING.md` — практический процесс изменения: Change Gate, проверки,
  rollback/canary и правила PR.

Минимальные проверки документационной и контрактной целостности:

```bash
./tools/surface-lint.sh
./tools/holo-verify.sh
```

Базовая flake-проверка:

```bash
nix flake check
```

## Рабочий цикл

Обычный цикл изменения выглядит так:

1. `inspect`: прочитать `AGENTS.md`, `SURFACE.md`, `HOLO.md`, релевантные модули
   и существующие Proof.
2. `contract`: определить, меняется ли публичная поверхность. Для FROZEN
   поверхности сначала зафиксировать Proof и Migration.
3. `patch`: внести минимальную правку в код или документацию.
4. `verify`: запустить релевантные проверки из `SURFACE.md`, `HOLO.md` и
   `CONTRIBUTING.md`.
5. `switch`: применять live-конфигурацию только после preflight-проверок.

Подробный процесс для авторов изменений находится в `CONTRIBUTING.md`.

## Границы репозитория

В репозитории хранится:

- декларативная NixOS- и Home Manager-политика;
- публичные контракты и решения;
- проверяемые scripts/tools/tests;
- воспроизводимые entrypoint-ы для Emacs, ops, agents и LLM.

В репозитории не должны храниться:

- приватные ключи, токены, пароли и незашифрованные credentials;
- machine-local state, который нельзя воспроизвести из конфигурации;
- временные build artifacts и ручные результаты диагностики;
- host-secret данные без явной схемы шифрования и операционного владельца.

Host-specific секреты и локальные исключения должны жить вне публичной истории
или в operator-managed encrypted artifacts.

## Короткая формула

`pro-nix` — это не набор dotfiles и не один NixOS-конфиг. Это
воспроизводимая рабочая система, где ОС, Emacs, сеть, ops-инструменты, агенты и
проверки связаны единым контрактом: что обещано, то описано; что описано, то
проверяется; что проверяется, то можно безопасно развивать.
